     Fix: Live ISO Install Process Broken ‚ÄĒ Boots Straight to Unprivileged CLI

     Context

     After booting the shaughvOS ISO in VirtualBox and selecting "Install
     shaughvOS" from GRUB, the system skips the Calamares graphical installer
     entirely and drops the user into an unprivileged CLI session where they
     can't even run sudo desktop on. No installation procedure occurs.

     The secondary issue: VirtualBox can't auto-detect the OS type from the ISO
     (requires manual "Debian 64-bit" selection). This is cosmetic and normal
     for custom ISOs.

     Root Cause Analysis

     I traced the full boot chain: GRUB ‚Üí live-boot ‚Üí systemd ‚Üí lightdm ‚Üí login
     scripts. There are 4 cascading failures in the imager's ISO creation path
     (.build/images/shaughvos-imager, lines 629-844):

     1. .install_stage = 10 ‚ÄĒ Login script enters error mode (CRITICAL)

     The build process (shaughvos-build) sets /boot/shaughvos/.install_stage to
     10 as a "ready for first-boot" marker. Normally, shaughvos-firstboot runs
     on first boot and resets it to 0, kicking off the update ‚Üí software install
      ‚Üí 2 (complete) chain.

     But the imager disables shaughvos-firstboot (line 688) for the live
     environment ‚ÄĒ correctly, since we don't want first-run setup in a live
     session. However, it never sets .install_stage = 2.

     Result: Every interactive shell triggers shaughvos-login (via
     /etc/bashrc.d/shaughvos-legacy.bash), which sees .install_stage = 10,
     doesn't match 2 (normal) or [01] (first-run), and falls into
     Prompt_on_Failure() ‚ÄĒ showing error dialogs about "unknown install state"
     and trying to restart first-run setup.

     File: shaughvos/shaughvos-login:247-284

     2. No NOPASSWD sudo for admin user (CRITICAL)

     The imager sets admin:1234 as the password (line 692) and configures
     lightdm autologin for admin (lines 696-700). But it never creates a sudoers
      rule for admin.

     The Calamares desktop entry (assets/calamares/calamares-installer.desktop)
     runs Exec=sudo calamares with Terminal=false. Without NOPASSWD sudo:
     - XDG autostart ‚Üí sudo calamares ‚Üí no terminal for password prompt ‚Üí silent
      failure
     - CLI ‚Üí sudo desktop on ‚Üí prompted for password (admin doesn't know to use
     1234)
     - CLI ‚Üí desktop on ‚Üí G_CHECK_ROOT_USER fails (not root)

     3. shaughvos-postboot still enabled (MODERATE)

     The imager disables shaughvos-firstboot, shaughvos-preboot,
     shaughvos-fs_partition_resize, and shaughvos-ramlog ‚ÄĒ but leaves
     shaughvos-postboot enabled. Postboot runs scripts from
     /boot/shaughvos/postboot which may attempt operations that don't make sense
      in a live environment (network config, service management, etc.).

     4. GRUB "Install" vs "Live" entries are functionally identical (MINOR)

     Both entries boot boot=live quiet splash ‚ÄĒ the only difference is nomodeset
      on the "Live" entry. Neither triggers Calamares any differently. This is
     fine since Calamares auto-launches via XDG autostart on desktop login, but
     it's confusing that both options do the same thing.

     What the User Experiences

     1. Select "Install shaughvOS" from GRUB
     2. live-boot mounts squashfs overlay
     3. lightdm starts ‚Üí autologins admin ‚Üí Xfce desktop appears (Xfce IS
     installed in the base image)
     4. XDG autostart tries sudo calamares ‚Üí fails silently (no NOPASSWD sudo,
     no terminal)
     5. User sees desktop but no Calamares installer window
     6. User opens terminal or gets dropped to CLI ‚Üí shaughvos-login triggers
     error dialogs
     7. User can't run desktop on or sudo anything meaningfully

     Fixes

     All changes are in .build/images/shaughvos-imager, within the ISO creation
     path (Step 5b area, after line 693).

     Fix 1: Set .install_stage = 2 for the live session

     After disabling first-boot services, tell the login script that setup is
     complete:

     # Tell login script setup is complete ‚ÄĒ skip first-run flow in live session
     echo 2 > "$live_root/boot/shaughvos/.install_stage"

     Insert after line 693 (after chroot "$live_root" bash -c "echo 'root:1234' 
     | chpasswd")

     Fix 2: Add NOPASSWD sudo for admin

     Create a sudoers drop-in so admin can run Calamares and other admin
     commands:

     # Grant admin passwordless sudo for the live session (required for 
     Calamares)
     echo 'admin ALL=(ALL:ALL) NOPASSWD: ALL' >
     "$live_root/etc/sudoers.d/live-admin"
     chmod 0440 "$live_root/etc/sudoers.d/live-admin"

     Insert after Fix 1

     Fix 3: Disable shaughvos-postboot in live environment

     chroot "$live_root" systemctl disable shaughvos-postboot 2>/dev/null ||
     true

     Add to the existing service disable block (after line 690)

     Fix 4: Set desktop autostart index

     Ensure the autostart system knows we want desktop mode:

     echo 2 > "$live_root/boot/shaughvos/.shaughvos-autostart_index"

     Insert after Fix 2 (ensures desktop status reports correctly and autostart
     works if login script path is taken)

     Files to Modify

     - .build/images/shaughvos-imager ‚ÄĒ Add ~6 lines to the "Step 5b" section
     (live-boot preparation, around lines 682-701)

     After rebuilding the ISO:
     1. Boot in VirtualBox (manually select "Debian 64-bit" as OS type)
     2. Select "Install shaughvOS" from GRUB
     3. Xfce desktop should appear with Calamares installer auto-launching
     4. Calamares should show the welcome ‚Üí locale ‚Üí keyboard ‚Üí partition ‚Üí
     users ‚Üí summary workflow
     5. Opening a terminal should show normal shaughvOS banner (not error

     Fix: Live ISO Still Boots to CLI ‚ÄĒ lightdm Not Installed + desktop Not 
     Executable

     Context

     After deploying v1.8.2 through v1.8.4, the live ISO still boots straight to
      a CLI login instead of the Xfce desktop with Calamares. The previous fixes
      addressed real bugs (.install_stage, sudo, nomodeset, squashfs /boot
     exclusion), but the PRIMARY blocker remains: lightdm was never installed in
      the ISO, so there's no display manager to start the graphical session.

     Additionally, the desktop command has a "Permission denied" error because
     it's tracked as non-executable (100644) in git ‚ÄĒ a Windows artifact from
     when it was created.

     Root Cause Analysis

     1. lightdm is NOT in the base image (CRITICAL ‚ÄĒ the blocker)

     Traced the full software install chain:
     - shaughvos-software installs Xfce (ID 25) at line 3338: G_AGI xfce4 
     xfce4-terminal gnome-icon-theme tango-icon-theme upower polkitd
     - This does NOT install lightdm. Only LXDE (ID 23) pulls in lightdm as a
     dependency.
     - The base image uses root autologin on tty1 ‚Üí shaughvos-login ‚Üí
     Run_AutoStart(2) ‚Üí exec startx ‚ÄĒ no lightdm needed for root.
     - The imager runs systemctl enable lightdm in the chroot (line 709), but
     lightdm isn't installed ‚ÄĒ this fails silently due to || true.
     - Result: No display manager ‚Üí no graphical login ‚Üí user gets a bare tty1
     text login.

     File: .build/images/shaughvos-imager:660 (apt-get install line)

     2. rootfs/usr/local/bin/desktop is non-executable (MODERATE)

     Git tracks the file as 100644 (no execute bit). Confirmed via git ls-tree 
     HEAD rootfs/usr/local/bin/:
     100644 blob cc3aa3c3d...  rootfs/usr/local/bin/desktop

     This is a Windows artifact ‚ÄĒ the Write tool creates files without Unix
     execute bits. When the rootfs overlay is deployed, desktop lands on the
     filesystem without +x. Result: "Permission denied" when any user tries to
     run it.

     File: rootfs/usr/local/bin/desktop

     3. Calamares install-to-disk flow (user's verification concern)

     The user wants assurance that Calamares will properly partition, format,
     install, and set up GRUB so the VM boots from disk after ISO removal. Code
     review of the Calamares modules:

     - partition.conf ‚ÄĒ GPT, ext4, 512M EFI partition. ‚úď
     - unpackfs.conf ‚ÄĒ extracts /run/live/medium/live/filesystem.squashfs to
     target. ‚úď (squashfs now includes /boot since v1.8.3)
     - fstab module ‚ÄĒ regenerates correct fstab for target partitions. ‚úď
     - bootloader.conf ‚ÄĒ grub-install with efiBootloaderId: shaughvos. ‚úď
     - GRUB packages ‚ÄĒ the installer (shaughvos-installer:1066-1073) installs
     grub-pc for BIOS or grub-efi-amd64 for UEFI. These are in the base image. ‚úď
     - services-systemd.conf ‚ÄĒ re-enables preboot/postboot/ramlog. ‚úď (added in
     v1.8.3)
     - shellprocess.conf ‚ÄĒ removes live-only sudo rule and autologin. ‚úď (added
     in v1.8.3)
     - users.conf ‚ÄĒ creates user with autologin, sets root password. ‚úď

     The Calamares flow is sound IF Calamares actually launches ‚ÄĒ which it can't
      until lightdm is fixed.

     Fixes

     Fix 1: Install lightdm in the imager's apt-get line

     File: .build/images/shaughvos-imager, line 660

     Change:
