     shaughvOS — Complete Build Plan

     Context

     shaughvOS is Emmett's custom operating system, forked from
     https://github.com/MichaIng/DietPi (lightweight Debian). The repo lives at
     RealEmmettS/shaughvOS — this is an independent fork, not an upstream
     contribution. The end goal: zero DietPi references anywhere in the codebase
      (except an attribution section in the README). The OS ships with QubeTX
     300 Series diagnostic tools, a themed Xfce+Dracula desktop, a Plymouth boot
      splash, and a polished terminal experience. The final deliverable is a
     flashable .img / bootable .iso that can be written to an SD card, USB, or
     loaded into a VM.

     Current state: 332 files, 155 of which contain DietPi references (8,662
     total occurrences). No rebrand work has been done yet — the codebase is
     essentially stock DietPi v10.2 RC3 plus branding assets (logo, wallpaper,
     fonts) and documentation.

     ---
     Phase 0: Prep & Infrastructure

     0.1 — Create dev branch

     - All work happens on dev, merged to master only when stable
     - Verify origin points to RealEmmettS/shaughvOS (confirmed)

     0.2 — Add rebrand audit CI job

     - New workflow: .github/workflows/rebrand-audit.yml
     - Counts all dietpi (case-insensitive) occurrences in repo
     - Reports count in CI output — this is our progress meter
     - Allowlist: README.md attribution section, CHANGELOG.txt historical
     entries, .update/patches historical migration code

     0.3 — Identify free software IDs

     - Scan dietpi/dietpi-software for the highest used ID and find 3
     consecutive free IDs
     - Allocate: TR-300 = 218, ND-300 = 219, SD-300 = 220 (or whatever is free)

     ---
     Phase 1: QubeTX 300 Series Integration

     Files: dietpi/dietpi-software, dietpi.txt

     1.1 — Register software metadata

     In dietpi-software registration section (~line 1213), add for each tool:
     - aSOFTWARE_NAME, aSOFTWARE_DESC, aSOFTWARE_CATX=8 (System Stats &
     Management)
     - aSOFTWARE_DOCS → QubeTX docs URL
     - Arch restrictions: block armv6l (1), armv7l (2), riscv64 (11)

     1.2 — Install blocks

     Follow the File Browser (ID 198) pattern exactly:
     - Arch case-switch: G_HW_ARCH 3 → aarch64-unknown-linux-gnu, * →
     x86_64-unknown-linux-gnu
     - Fetch latest release tag from GitHub API
     - Download_Install to /usr/local/bin/
     - chmod +x on the binary
     - ND-300 installs both nd300 and speedqx from same release

     1.3 — Uninstall blocks

     Simple file removal: [[ -f '/usr/local/bin/tr300' ]] && G_EXEC rm 
     /usr/local/bin/tr300 (repeat for each binary)

     1.4 — dietpi.txt defaults

     - AUTO_SETUP_NET_HOSTNAME=shaughvOS
     - AUTO_SETUP_GLOBAL_PASSWORD=shaughvos
     - AUTO_SETUP_INSTALL_SOFTWARE_ID=218 219 220 25 (QubeTX + Xfce)
     - AUTO_SETUP_AUTOMATED=1
     - AUTO_SETUP_AUTOSTART_TARGET_INDEX=2 (desktop autologin)
     - SURVEY_OPTED_IN=0

     ---
     Phase 2: Desktop Environment

     2.1 — Dracula theme setup

     Add to the Xfce post-install flow in dietpi-software:
     - Download Dracula GTK theme → /usr/share/themes/Dracula/
     - Download Papirus-Dark icons → /usr/share/icons/Papirus-Dark/
     - Configure via xfconf-query or Xfce XML config files in .conf/desktop/
     - Set Dracula WM theme, GTK theme, icon theme

     2.2 — Font installation

     - Copy assets/fonts/Makira/*.ttf → /usr/share/fonts/truetype/shaughvos/
     - Copy assets/fonts/IBMPlexMono/*.ttf →
     /usr/share/fonts/truetype/shaughvos/
     - Run fc-cache -f
     - Set Makira as default GTK/UI font
     - Set IBM Plex Mono as terminal font in Xfce Terminal config

     2.3 — Wallpaper

     - Copy assets/desktop_background.jpg → /usr/share/backgrounds/shaughvos/
     - Set as default Xfce desktop wallpaper

     2.4 — Panel layout

     - Bottom taskbar with app menu, system tray (Zorin/Windows-like layout)
     - Configure via Xfce panel XML config

     2.5 — desktop toggle command

     New file: rootfs/usr/local/bin/desktop
     desktop on     → autostart index 2, systemctl start lightdm
     desktop off    → autostart index 7, systemctl stop lightdm
     desktop status → show current mode
     --help         → usage info
     Sources shaughvos-globals, checks root, provides status feedback.

     ---
     Phase 3: Boot & Terminal Experience

     3.1 — Plymouth boot splash

     New directory: rootfs/usr/share/plymouth/themes/shaughvos/
     - Render white-on-black PNG of assets/shaughv-logo.svg (256x256, 512x512)
     - Create shaughvos.plymouth config + shaughvos.script
     - Add plymouth + plymouth-themes to install deps
     - Set as default: plymouth-set-default-theme shaughvos && update-initramfs 
     -u

     3.2 — Terminal splash + TR-300 auto-run

     New file: rootfs/etc/bashrc.d/shaughvos.bash
     1. Display ASCII art shaughvOS banner (embedded or from
     /boot/shaughvos/.ascii_banner)
     2. Run tr300 --fast if binary exists
     3. Set alias report='tr300'

     Phase 9: OTA Update Infrastructure

     How OTA updates already work

     The update system (shaughvos-update) is entirely GitHub-based:

     1. Device reads DEV_GITOWNER and DEV_GITBRANCH from /boot/shaughvos.txt
     (defaults: RealEmmettS, master)
     2. Fetches
     https://raw.githubusercontent.com/$OWNER/shaughvOS/$BRANCH/.update/version
     to check for new version
     3. If newer version available, downloads pre-patches, pulls new code from
     GitHub, runs patches
     4. No custom server needed — GitHub raw content IS the update server

     9.1 — Configure OTA defaults

     In shaughvos-update (line 49 equivalent):
     GITOWNER_TARGET=${GITOWNER_TARGET:-RealEmmettS}
     GITBRANCH_TARGET=${GITBRANCH_TARGET:-master}

     In shaughvos.txt:
     DEV_GITOWNER=RealEmmettS
     DEV_GITBRANCH=master

     All raw.githubusercontent.com URLs change from $OWNER/DietPi/$BRANCH to
     $OWNER/shaughvOS/$BRANCH.

     9.2 — Release → OTA flow

     When we want to push an update to all deployed devices:

     1. Develop on dev branch — all work happens here
     2. Merge dev → master — this is the "release"
     3. Update .update/version — bump G_REMOTE_VERSION_SUB (or _CORE for major)
     4. Push to master — immediately available to all devices
     5. Devices check for updates on boot (via shaughvos-update) or manually
     6. They see the new version in .update/version, pull the code, apply
     patches

     Tag-based releases (for images): git tag v1.1.0 && git push --tags triggers
      the image build workflow. This produces downloadable images for new
     installs, while the master branch update serves existing devices via OTA.

     9.3 — Version bumping workflow

     .github/workflows/version-bump.yml (new, optional):
     - Trigger: workflow_dispatch with input for version type
     (major/minor/patch)
     - Automatically updates .update/version with new version numbers
     - Creates a commit and tag
     - Triggers the release-images workflow

     9.4 — OTA update notification (already built in)

     The existing shaughvos-banner (login banner) checks
     /run/shaughvos/.update_available and shows a notification when an update is
      pending. This Just Works once paths are rebranded.

     9.5 — Live patches (hotfixes without full update)

     The .update/version file supports G_LIVE_PATCH_DESC, G_LIVE_PATCH_COND, and
      G_LIVE_PATCH arrays for urgent fixes that apply without a full update
     cycle. We preserve this mechanism — push a live patch to master by editing
     .update/version and deployed devices pick it up on next boot.
